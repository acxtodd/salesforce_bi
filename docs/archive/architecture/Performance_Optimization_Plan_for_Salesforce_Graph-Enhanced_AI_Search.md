# **Performance Optimization Plan for Salesforce Graph-Enhanced AI Search**


## **Introduction**

The Salesforce-integrated AI search system currently faces severe latency (often **30+ seconds** for complex queries) due to graph traversal and multi-step data retrieval . The goal is to reduce end-to-end search latency below 5 seconds (ideally 1–2s) **without sacrificing result quality**. This plan outlines a comprehensive set of optimizations – spanning code efficiency, data modeling, infrastructure tuning, and caching – to achieve near real-time performance. We prioritize each recommendation by expected impact and implementation cost, leveraging best practices and the system’s known issues (e.g. cross-object traversal slowness, lack of caching) . By applying these changes, the retrieval pipeline will become far more efficient, ensuring a fast and seamless search experience.

## **Current Bottlenecks Overview**

**Profiling the Retrieve Lambda (3600 lines, Python 3.11)** reveals several bottlenecks contributing to ~30s latency:

* **Multi-hop Graph Traversal:** The cross_object_handler performs breadth-first searches (BFS) through DynamoDB-stored graph edges. Each hop triggers many sequential DynamoDB **query** and **getItem** calls. Although edges are indexed by fromId and toId (adjacency list pattern) for direct lookups , the traversal still issues *per-neighbor* getItem calls and iterates through multiple hops sequentially. This is the primary hotspot (often 20s+ alone for complex queries). The traversal isn’t always **short-circuited** efficiently  – e.g. it may continue exploring even after finding sufficient results (“seeds”), causing unnecessary work.

* **DynamoDB Filter Scans:** Filtering by related-object attributes (e.g. “City = Plano” on properties) currently involves scanning up to 5000 items from the graph_nodes table per query  . The code uses a GSI on type to fetch all nodes of a given object type (e.g. all properties), then applies the attribute filter in Python . This approach can scan thousands of items per query, especially for common fields like city or class, adding hundreds of milliseconds or more for large datasets. It effectively trades *index complexity* for simplicity, but at the cost of latency.

* **Redundant Sequential Operations:** The current orchestration performs many steps back-to-back where parallelism is possible. For example, after obtaining initial “seed” IDs via the graph filter, the system performs a Bedrock vector search **sequentially**, then an enrichment step, then answer generation . There is no caching of intermediate results (e.g. once a property’s availabilities are known, they are fetched from DynamoDB repeatedly for similar queries). Also, not all LLM planner calls or vector searches are cached, meaning repeated queries (even identical ones) incur full cost each time. Lack of reuse results in duplicated work.

* **Lambda Compute Constraints:** The Retrieve Lambda currently runs on **1024 MB (≈0.6 vCPU)** and x86_64 architecture. This limits its CPU throughput for data processing and the number of concurrent threads it can effectively run. As AWS notes, allocated memory directly governs available CPU – more memory means more vCPU and faster execution . At present, the function may be CPU-bound when merging results or formatting responses, and I/O-bound when waiting on DynamoDB. The **cold-start** overhead is also non-negligible (initializing 3.6k lines of code and libraries), potentially a few hundred milliseconds on x86_64. With no provisioned concurrency, occasional cold starts further hurt P95 latencies.

* **No External Caching Layer:** Every query hits DynamoDB and Bedrock fresh. The system does not use Redis/ElastiCache (as noted, none is currently deployed) to cache frequent query results or graph lookups. Thus, repeated queries (e.g. many users asking for “Class A offices in Plano”) each redo the expensive planning, scanning, traversal, and Bedrock retrieval work. This is a missed opportunity since many queries – especially in CRE – are recurrent (cities, property types, etc.). Caching could cut those query times to sub-second, but is not leveraged yet.

* **Underutilization of Derived Views:** The design includes denormalized **derived view tables** (availability_view, vacancy_view, etc.) intended to store pre-joined records for common cross-object queries. However, these views were not fully backfilled or kept updated (they were “empty pending clean backfill” as of the latest status) . As a result, the Retrieve Lambda isn’t currently using them for query execution. Instead of a single indexed lookup in a view, the system falls back to runtime graph traversal, incurring extra latency. This is a clear area for improvement since properly populated views can answer some multi-object queries in one shot.

With these pain points identified, we move on to the optimization strategies. Each recommendation is backed by rationale and, where possible, by measured best practices or relevant analogies from AWS architecture. The plan is organized from **quick, high-impact wins** to more **involved structural changes**.

## **Code-Level Optimizations (Graph Traversal & Query Execution)**

**1. Batch and Parallelize DynamoDB Operations:** The cross-object BFS is currently making DynamoDB calls in a very linear fashion – querying edges for one node at a time, then calling get_item for each adjacent node sequentially  . We can drastically cut this overhead by using **batch requests and parallelism**. Specifically:

* *Batch GetItem for Neighbors:* Modify _get_connected_nodes to collect all toId (outbound) and fromId (inbound) neighbor IDs first, then issue a single BatchGetItem to fetch all their node details in one round-trip. DynamoDB’s batch APIs allow up to 100 items per call, which aligns with the system’s MAX_NODES_PER_HOP = 100 limit . This replaces up to 100 individual get_item calls with **1 batched call**, significantly reducing network overhead. For example, if a property has 50 related availabilities, the current code does 50 synchronous calls; a batch get would retrieve all 50 in one go. This yields a near 50× improvement for that step.

* *Asynchronous Neighbor Fetching:* Introduce concurrency when traversing multiple seed nodes or exploring multiple branches at a given BFS depth. Python’s concurrent.futures.ThreadPoolExecutor or asyncio can be used to overlap DynamoDB I/O since boto3 releases the GIL during network calls. The BFS loop can enqueue neighbor-fetch tasks in parallel instead of waiting for each to complete serially. With the Lambda’s CPU allocation increased (see **Infrastructure** section), it can handle several parallel requests. This means if 10 property nodes each need to fetch edges, those 10 queries can happen simultaneously (in practice, bounded by DynamoDB throughput and Python thread overhead). The result is that multi-seed traversals complete in roughly the time of one seed’s traversal. Overall, batching and parallel I/O should reduce multi-hop traversal latency from several seconds to a few hundred milliseconds in many cases.

* *Early Termination (Short-Circuiting):* Ensure the traversal stops as soon as the target records are found in sufficient quantity. The code already limits collecting more than 100 target IDs , but it also should avoid scanning excess graph segments once that threshold is reached. We will audit the logic so that if, for example, **3 seeds yield 100+ results quickly, the function returns immediately** rather than continuing to examine remaining seeds or deeper hops. According to project notes, traversal was *“not always short-circuited when seeds exist”*  – fixing this bug will avoid wasted work. Essentially, once we have enough matches, we cut off BFS exploration promptly.

* *Use Relationship Hints:* The planner (LLM) often provides a **traversal_path** (e.g. Property → Availability) for cross-object queries. We can use this hint to restrict BFS to relevant edge types only. For instance, if the query is “availabilities at properties in Plano,” we know we only need to go from Property to Availability (child edge). Instead of retrieving *all* neighbors (which could include unrelated objects like leases or accounts), the code can specifically query the edges that lead to Availability records. This can be done if edges carry a type or if separate edge tables exist per relationship. In the current design, edges are in one table but we have a map of known relationships . We can filter the connected nodes to skip types that are not on the path to the target, avoiding futile traversal down irrelevant branches. This optimization prunes the search space and saves time, especially in a graph with multiple object types (the BFS won’t wander into contacts or cases when looking for availabilities).

* *Leverage Direct Keys:* In some one-hop cases, we might bypass graph traversal entirely by using foreign keys. For example, if the query is *“leases where Property city = X”*, the plan would find property IDs by city, then traverse to leases. But since each Lease node likely stores a property_id attribute, we could directly filter the leases by that property_id set (a direct DynamoDB Query on a GSI if available, or a BatchGet of those leases). The current approach always goes through the edges table, but for parent-child relationships that mirror a simple foreign key, a direct query can be faster. This is a micro-optimization to consider when implementing the traversal logic: detect if the next hop can be resolved via an attribute on the child rather than an edges lookup. (This is feasible because the graph builder populates graph_nodes with key attributes – e.g., Availability nodes might include their property_id in the attributes – so one could query the nodes table for those IDs directly).

**2. Optimize In-Process Filtering:** The _query_nodes_by_attributes method scans a slice of the nodes and then filters in Python for attribute matches  . We can make two improvements here:

* *Use DynamoDB FilterExpression for simple predicates:* For equality conditions and straightforward numeric comparisons, adding a FilterExpression to the DynamoDB Query can offload filtering to the database engine. For example, instead of retrieving 5000 property items and checking attributes["City"] == "Plano" in code, we can query the GSI with KeyCondition type = 'Property' and FilterExpression City = :city. DynamoDB will still scan the items internally, but it will only return matching items to the Lambda. This reduces data transfer and parsing in the Lambda runtime. It won’t drastically cut Dynamo read time (since non-indexed filters still cost throughput), but it streamlines processing. We must ensure case-insensitivity (the code currently lowercases strings for comparison ) – a solution is to store a normalized lowercase city in the item or use BeginsWith for case-insensitive match. With careful implementation, using filters could trim a few hundred milliseconds and simplify code.

* *Limit scans by smarter indexing:* The 5000-item scan limit (CROSS_OBJECT_QUERY_LIMIT) is a blind cap. In scenarios where matches are rare, we’re scanning far more than needed; conversely, if matches are common, 5000 might not retrieve all matches. A better strategy is to index high-cardinality fields. This is covered in the data model section below (creating GSIs for fields like City, Class). In code, once such indexes exist, _query_nodes_by_attributes can query the field-index directly (e.g. a GSI on City), eliminating the need for any server-side scan or client-side filter. This is a significant change, but worth mentioning here as it directly affects this function’s performance: it would turn a scan of 5000 into a direct lookup of ~N results.

* *Reduce Python overhead:* Ensure that our filtering loops use efficient structures (e.g. set lookups for membership) and avoid unnecessary conversions. The current code already does a decent job (e.g. converting Decimal to float once, lowercasing strings once) . Still, we should avoid O(n*m) patterns (though 5000× few filters is fine). This is a minor tweak: the big wins are from not having to filter so many items in the first place.

**3. Streamline the Enrichment Step:** After obtaining the target records (e.g. a list of Availability IDs), the Retrieve Lambda enriches them with related properties or other metadata . Currently, this likely involves a loop fetching each parent record (e.g. for each Availability, do a get_item on its Property ID to get name, class, etc.). We can optimize this exactly as we did for BFS neighbor fetches:

* *Batch fetch related records:* Collect all unique parent IDs needed for enrichment and perform a single BatchGet from DynamoDB (or a few batches if the set is large). For example, if 20 availabilities need enrichment, do **one** call to retrieve all 20 parent Property nodes instead of 20 calls. This yields an order-of-magnitude improvement for the enrichment stage.

* *Cache frequently used metadata:* Property details like name, city, class are relatively static. We can introduce an in-memory cache or utilize the existing schema_cache/vocab_cache mechanism to store recently accessed object metadata. For instance, if the same property IDs appear in multiple queries, we can store the property info in a Python dict within the Lambda (which persists across invocations when warm) or in an external cache. This way, enrichment can first check the cache and skip a DynamoDB call if the data is already known. Given that some properties (e.g. popular buildings or accounts) will recur across queries, a simple LRU cache with a TTL can save repeated lookups.

* *Use Derived View (if available):* In the long term, if the **availability_view** table is fully populated with property fields (as designed  ), the enrichment step becomes unnecessary for availability queries – the availability records would already *contain* the property’s city, class, etc. Therefore, one strategic recommendation is to rely on those denormalized views to deliver pre-enriched records (more on this in the next section). This not only saves time in the Retrieve Lambda, but also simplifies the code (fewer moving parts at query time).

By applying the above code-level optimizations, we expect to **drastically reduce the graph traversal and enrichment latencies**. The BFS will make far fewer round-trips to DynamoDB and leverage parallel execution, converting a previously ~20–25s traversal into perhaps ~1–2s or less. These changes are mainly code and configuration adjustments, not requiring new infrastructure – thus they offer a **high benefit/cost ratio**.

## **Data Modeling & Precomputation Improvements**

**4. Leverage Denormalized Derived Views:** The system already anticipates the performance limitations of runtime graph joins by providing *pre-joined* DynamoDB tables (like availability_view, vacancy_view, leases_view) that aggregate fields across related objects . The strategy here is to **populate and utilize these views for query retrieval** instead of on-the-fly graph traversal whenever possible. Key actions:

* **Complete the Backfill:** First, ensure all relevant records are loaded into the derived view tables. The architecture notes indicate that some views were empty or stale (e.g. only vacancy_view had data, others were pending backfill as of Dec 2025) . Running the backfill Lambda (and fixing any issues in the enrichment pipeline that populates these views) is crucial. Once these tables (e.g. availability_view) accurately mirror the latest data (and continue to update via CDC streams), they become a powerful tool: effectively a **materialized join** of Property + Availability, Property + Lease, etc.

* **Query Views Instead of Graph**: Teach the Planner or Retrieve logic to route certain queries to the view. For example, a query like “show Class A office availabilities in Plano” can be answered by querying the availability_view table for items where property_class = 'A' and city = 'Plano' and property_type = 'Office'. In one DynamoDB Query, using a suitable index, we would get all matching Availability records (with their property info already attached). This would replace steps 1–4 of the current pipeline with a single fast lookup. **No multi-hop traversal, no separate enrichment step** – the view data is already unified. We expect this to cut latency from a few seconds to tens of milliseconds for those queries. It also reduces Bedrock vector usage: if the view item contains or references the textual content needed for the LLM answer, we might not even need a vector search for structured queries. (Alternatively, we could store a reference to a KB chunk ID in the view, and directly fetch those specific chunks.)

* **Index the Views for Common Filters:** To maximize efficiency, we should add Global Secondary Indexes on the view tables that cover typical filter fields. For instance, on availability_view, define a composite GSI like (city, property_class) or separate GSIs for city and for property_class if queries often filter by them independently. DynamoDB can handle a high cardinality attribute like city as a partition key (each city’s data will be sharded). With such GSI, the query *“City=Plano AND Class=A AND Type=Office”* can be served by querying Index = city-index, KeyCondition city='Plano' with a FilterExpression for class=‘A’ and type=‘Office’, or by a more complex composite key if we anticipate combinations. The exact indexing strategy can be informed by query logs (which filters are most frequent). This index design is critical – without it, we’d have to Scan the availability_view (which could be large). The goal is a direct lookup. In summary, **denormalize then index** for the query patterns we know: this is a textbook approach to speeding up read-heavy workloads at the expense of extra storage and write cost.

* **Maintain and Monitor Freshness:** Using derived views means we rely on an asynchronous pipeline (ingest enrichments via Lambda/Step Functions) to keep them up-to-date. We must ensure the CDC (Change Data Capture) from Salesforce is reliable and low-latency so that new or updated records flow into the views quickly. In practice, slight staleness (on the order of a few minutes) might be acceptable for search, but we should configure monitoring to detect any backlog or failures in populating the views. If a view is incomplete, the planner should perhaps fall back to the graph traversal method (to avoid false negatives). Over time, as confidence in the view’s completeness grows, we can rely on it solely. This will allow retiring some of the runtime traversal logic for those query types.

Using derived views aligns perfectly with our performance goals: **precompute expensive joins** so that user queries don’t have to. It’s a common best practice for speeding up relational queries in a NoSQL environment – essentially moving compute from query-time to ingest-time. The expected impact is huge: cross-object queries that once took 2–5s (in an optimized traversal) could answer in **<100ms** via a single table lookup. This addresses the known limitation that *“cross-object queries add latency (2–5s vs sub-second for direct queries)”* . By making them “direct” (pre-joined), we achieve that sub-second performance.

**5. DynamoDB Indexes & Access Patterns:** In cases where a full derived view is not feasible, we should refine the DynamoDB data model to better support the queries. Some suggestions:

* **Global Secondary Index per frequent filter**: Identify the top filters used in queries (e.g. City, State, Industry, RecordType) and add GSIs to the graph_nodes table on those attributes. For example, a GSI on City (partition key = city, sort key = createdAt or nodeId) would allow querying all properties in “Plano” directly, rather than scanning by type. Another on RecordType might help if users often filter by record type (e.g. Office vs Industrial). Each GSI will incur storage and write overhead (DynamoDB writes will now also update the index), so we prioritize those that yield significant read savings. This is a trade-off: adding 2–3 well-chosen GSIs can drastically cut read latency for corresponding queries. **Note:** If combining filters (City AND Class) is common, we could use a composite key (e.g. partition = city, sort = class#nodeId) or create a multi-attribute GSI. Dynamo doesn’t support arbitrary multi-attribute queries without such composites, so we design for known combinations that matter.

* **Adjacency List Enhancements:** The current graph modeling uses an *edges table* keyed by fromId with a toId-index for reverse lookups . This is already a good practice (adjacency list pattern)  . We might consider *projecting* some node attributes onto edge items to reduce the need for node fetches. For example, each edge item could carry the type of the toId node, or even a small subset of its attributes (like name or status). That way, the BFS could identify target nodes without always hitting the nodes table. However, duplicating data in edges must be done cautiously to avoid consistency issues. Since we’ve decided to batch-get the node data, this edge projection is optional. Our primary approach (batch fetching nodes) should suffice, but this is a possible micro-optimization if some relationship is extremely hot and we want truly one-round-trip traversal.

* **Consider Item Collection Model:** Another schema alternative is to store related entities together as an item collection in a single table (using a composite sort key). For example, store a Property and its child Availabilities in the same partition (with sort keys distinguishing them). Then “availabilities by property” becomes a single Query on that partition. However, our cross-object queries often involve filtering by the child’s parent’s attributes (city, etc.), which this alone doesn’t solve without a GSI. Given that we already have separate tables and a working ingestion pipeline, a full single-table design refactor would be high effort. We focus on augmenting the current design with indexes and views rather than total redesign.

* **Amazon Neptune (Graph Database) for Multi-hop:** If the relational complexity grows or we need **millisecond** latency on more complex traversals, we should evaluate offloading the graph to a purpose-built graph database like **Amazon Neptune**. AWS explicitly recommends Neptune for use cases requiring “multiple node aggregations of second- and third-level relationships” with sub-millisecond performance . Neptune can store the CRE graph (properties, availabilities, leases, accounts, etc.) and answer multi-hop queries using Gremlin or SPARQL queries that are highly optimized. For instance, a Gremlin query can find all Availabilities at Properties in Plano in a single traversal operation on the DB side, likely much faster than our Lambda orchestrated approach once data size grows. **However**, adopting Neptune is a significant architectural change: it requires migrating or dual-writing the data, and the team would need to manage a new cluster. It also adds cost and operational overhead. Given the current size of data (perhaps manageable within DynamoDB) and the existence of derived views as an alternative, Neptune is a **long-term option** if we hit a wall. We include it in the plan for completeness (especially if future queries demand deeper hops or more graph analytics), but we prioritize optimizing DynamoDB and caching first.

In summary, our data-layer changes aim to eliminate expensive scans and run-time joins. By precomputing relationships (derived views) and carefully indexing DynamoDB for our query patterns, we turn the retrieval problem from a compute-heavy task into a simple key-value look-up problem, which DynamoDB excels at (single-digit millisecond responses on key queries). These changes, combined with the code optimizations, address the **core data access latency** that has been plaguing the system.

## **Caching Strategies**

Caching provides outsized benefits when the same or similar queries repeat, which is common in real-world usage. We propose introducing caching at multiple levels:

**6. Introduce an External Cache Layer (Redis/ElastiCache):** Given that **no Redis/ElastiCache is currently used** (but it’s allowed to add) and many search queries are repeated with minor variations, a distributed in-memory cache can drastically reduce latency for subsequent occurrences of a query. Key considerations:

* **What to Cache:** We can cache **final search results** or **intermediate results**. Caching the final LLM answer for a natural language query is tricky (due to user-specific permissions and potentially differing phrasing), so a better approach is to cache the retrieval results (the set of record IDs or the assembled context for the LLM) for a given structured query plan. For example, after the planner interprets “Class A office availabilities in Plano” into a filter (PropertyClass=A, City=Plano, RecordType=Office) , we use that structured plan (plus user auth context) as a cache key. The next time the same plan appears, we can directly reuse the list of availability IDs (or even the fetched documents) instead of recomputing them. Essentially, **cache the output of the Retrieve Lambda for identical plans**.

* **Key Design:** A suitable cache key could be a hash of {user_tenant}:{target_object}:{filters}. Including the user or their access scope is important because of Salesforce sharing rules – two users asking the same query might see different results. We might incorporate the user’s “sharing buckets” (the authz groups) into the key so that we don’t serve results a user isn’t entitled to. If that’s too fine-grained, caching per user role or profile could be an alternative (assuming similar access). The value stored would be the retrieval payload: e.g. the list of top-K record IDs and any snippet texts from the KB. We should also cache negative results (queries that returned no records) to avoid repeat heavy computation on hopeless queries.

* **TTL and Invalidation:** We’ll set a **time-to-live (TTL)** on cache entries (e.g. 5 minutes or 1 hour) based on how fresh the data needs to be. Since Salesforce data changes propagate through ingestion to DynamoDB, the cache should be invalidated on relevant data updates. A sophisticated approach is to integrate with the CDC pipeline: e.g. if a property’s city changes or a new availability is added, we could proactively invalidate cache entries that depend on “City=Plano” or that property. This requires tracking query dependencies, which may be complex. As a simpler approach, use short TTLs (e.g. 5-10 minutes) so cache naturally refreshes and stale data is limited. In practice, CRE data (availability, leases) doesn’t change every second, so even a 10-minute cache could be acceptable, especially if the LLM answer will be regenerated each time (ensuring current data is reflected eventually).

* **Size and Cost:** A managed Redis (Amazon ElastiCache for Redis or Memcached) will incur additional cost, but we can start small (say a cache node with a few GB of memory). The latency benefit is huge – cache hits would return in a few milliseconds over the network, versus seconds of computation. Even a 50% cache hit rate could cut the average response time dramatically. We should monitor usage patterns to tune the cache size and eviction policy (LRU by default). The cost of a cache is typically far lower than recomputing expensive queries multiple times, and it also offloads pressure from DynamoDB/Bedrock (potentially saving their throughput costs). This is a **medium effort, high reward** change.

* **Potential Cached Content:** In addition to query results, we can cache *planner outputs*. The Query Planner uses an LLM via Amazon Bedrock which likely takes a few hundred milliseconds to interpret a query. Many users might ask semantically identical questions (“availabilities in Plano that are Class A offices” vs “Class A office space in Plano” – the planner would produce the same filters). We can cache the StructuredPlan for recent queries (using the normalized filter dict as key) so that we bypass the LLM call on repeats. This not only improves latency (Bedrock call skipped) but also reduces LLM usage cost. Given that the schema and vocabulary caches are already in place for helping the planner, this is a natural extension: a simple in-memory dict or DynamoDB table mapping query text (or its key attributes) to the planner result. The *vocab_cache* may already do some of this for entity recognition; we’d implement a higher-level cache for the whole plan.

**7. Intra-Invocation Caching and Warm State:** Besides a shared external cache, we can exploit the fact that AWS Lambda execution environments are **reused** across multiple invocations (when warm). By using **Provisioned Concurrency** (see next section), we can keep instances alive and warm. This allows us to cache data in global variables for quick reuse within the same instance. Strategies include:

* **Preload Hot Data on Init:** On Lambda startup (global scope), we could load a small portion of the graph into memory. For example, load all Property nodes (if there are only a few thousand) into a dictionary keyed by propertyId, or load a mapping of City → [PropertyIDs]. This would make filtering queries extremely fast (no Dynamo calls needed for that part). Because the Lambda has 1024+ MB memory, holding a few MB of frequently queried data is feasible. We must be careful to only do this if it’s truly needed and beneficial (to avoid prolonging cold start). Alternatively, we lazy-load such data on first use and keep it around. For instance, after the first time we scan properties in Plano, cache that list in memory for subsequent queries in the same session.

* **Reuse Bedrock Results:** If Bedrock vector search results (chunks) are needed for multiple queries (perhaps overlapping content), a Lambda instance could keep a cache of recently retrieved chunks (by recordId or by query embedding). However, since our queries are quite distinct and we use record filters, this might be less useful. The external cache is better suited for cross-invocation sharing of any such data.

* **Function Warming:** We ensure that the global initialization (like establishing DynamoDB connections, loading small caches) runs once and then use it for subsequent invocations. The code already uses lazy singletons for DynamoDB table objects , which is good (it avoids re-initializing the boto3 client each time). We can extend this pattern to other resources (e.g. Bedrock SDK clients, authz cache loads, etc.). The idea is to **pay any setup cost only once** per container. This doesn’t reduce the 30s runtime directly, but it shaves cold-start overhead and ensures each request’s 35s budget is mostly spent on the real query logic.

**8. Cache Invalidation and Coherency:** With caching in place, we need to maintain accuracy. We’ve touched on TTLs and possibly event-driven invalidation. We should also implement a mechanism to bypass cache easily for debugging or ensure that stale caches are not used beyond acceptable limits. Monitoring cache hit rates and stale-return incidents will be important in the tuning phase. In practice, slight staleness might be acceptable for search (users typically won’t mind if a just-updated record appears a few minutes later in search results), especially since our Answer Lambda ultimately verifies data via citations. Nonetheless, designing a robust cache-coherency strategy (perhaps tying cache entries to a data version or timestamp) would be part of implementation best practices.

In summary, caching is about avoiding repeat work. Given the system’s current state (no caching at all), introducing it can yield **order-of-magnitude improvements** in many scenarios. A previously expensive query that is cached could return in <50ms (just a Redis fetch), which is *instantaneous from the user’s perspective*. Even for queries that aren’t exact repeats, partial caching (like reusing sub-results or planner outputs) will make the system more responsive under load.

## **Infrastructure & Configuration Tuning**

While code and data optimizations cut down the intrinsic work needed, infrastructure settings ensure the Lambda can perform that work as quickly as possible. We recommend:

**9. Increase Lambda Memory (and CPU) Allocation:** The Retrieve Lambda is currently at 1024 MB, which corresponds to roughly 0.6 vCPU. AWS Lambda proportionally allocates CPU with memory – at ~1769 MB you get a full vCPU, and up to 6 vCPUs at 10 GB memory  . By **doubling or tripling the memory (e.g. to 2048 MB or 3072 MB)**, we provide significantly more CPU cycles for our compute-heavy tasks (JSON processing, Python loops, etc.) and to better handle multi-threading. This often yields a **super-linear speedup**: more CPU shortens the duration, which can make costs nearly flat. AWS’s own example showed a 10× performance improvement (11.7s -> 1.46s) when going from 128 MB to 1024 MB, with virtually the same cost . In our case, moving to 2048 MB (≈1.2 vCPUs) or higher could cut pure Python execution time by 2–3× or more. Since parts of our workflow (like BFS filtering logic, merging results, serializing output) are CPU-bound, this directly accelerates them. It will also help handle concurrency: multiple threads can make better progress if more CPU is available. We will use AWS’s Lambda Power Tuning tool or empirical testing to find the sweet spot, but **initial evidence suggests that upping memory will decrease duration enough that the GB-sec cost stays similar or might even improve**  .

* *Cost Impact:* We must be mindful that increasing memory increases cost per millisecond, but since duration should decrease, the net effect can be minimal. For instance, if 1024 MB -> 2048 MB halves the execution time, cost remains roughly the same (because 2× memory * 0.5× time = 1× cost). If it reduces time further (as often happens with IO-bound functions benefiting from CPU), it could even save money. Thus, from a cost/benefit view, this is a low-hanging fruit to achieve latency goals.

**10. Migrate to AWS Graviton2 (arm64) Architecture:** Currently, production is on x86_64, but the team *“recommended”* migrating to arm64 , which is AWS’s Graviton2/3 processors. arm64 Lambdas often show **better performance for certain workloads and ~20% lower cost** compared to x86. Python applications generally run very well on Graviton; many scientific Python libs have ARM-compatible wheels now. We should thoroughly test the Retrieve and Answer Lambdas on arm64 (AWS makes this switch easy via configuration). Expected benefits include: slightly faster initialization, and possibly faster CPU throughput due to Graviton’s efficient architecture (especially if our code uses native libraries that are optimized for ARM). Additionally, the cost per 100ms is lower on arm, so any given workload is cheaper by default. This directly contributes to cost efficiency if we run provisioned concurrency or high volumes. Given the maturity of arm64 on Lambda, this change is **low-risk and high-reward**. We will need to ensure all dependencies are either pure Python or available for arm (boto3, etc. are fine; if any library isn’t, we find an alternative). The deployment process (CI/CD) should be updated to package for arm64. This should be done sooner rather than later, as it’s largely a one-time effort with ongoing benefit. The architecture document explicitly lists this as a future hardening step (arm64 migration) .

**11. Enable Provisioned Concurrency (and/or Adaptive Warming):** Provisioned Concurrency (PC) keeps a set number of Lambda instances initialized and ready to serve, eliminating cold starts. For a user-facing search with low tolerance for occasional 1-2s cold lag, PC is advisable. Specifically, we recommend provisioning a small pool of concurrent instances (based on expected QPS and burst). For example, if peak load is 5 concurrent queries, provision 5–6 instances. This guarantees **consistent response times**. Moreover, PC will allow us to take advantage of the warm-start caching discussed: an instance remains active and can retain caches in memory across many requests. Without PC, if traffic is spiky, Lambda might scale down to 0 and then cold start frequently, negating in-memory caches.

* *Trade-offs:* PC does incur a cost for idle time – you pay for the hours instances are provisioned regardless of usage (at a discounted rate). We can mitigate waste by using **Auto Scaling** on PC (AWS allows scheduled or usage-based adjustment). For instance, during business hours when search usage is high, keep N provisioned; off-hours, reduce it. The Search system likely aligns with business users (daytime usage), so this is feasible.

* The architecture note highlights *“provisioned concurrency; trim packages”* as a to-do . So the intention to use PC is already there. We will implement this in tandem with code trimming.

**12. Optimize Dependency Loading and Package Size:** Large Lambda package size and heavy init code can hurt both cold start and memory usage. We should audit our dependencies: remove any libraries that aren’t essential, and consider using Lambda layers or slimmer alternatives. For example, if we only use a subset of AWS SDK functionality, ensure we’re not bundling the entire SDK (in Node you can import specific clients; in Python, boto3 is already optimized and comes pre-installed in the runtime environment). Another example: if using Pandas or other heavy libraries just for minor tasks, see if we can drop them. The AWS Compute Blog suggests that reducing package size (especially cutting out unused parts of AWS SDK, X-Ray, etc.) can save 100–200ms on init  . In our case, the Retrieve lambda likely includes the LLM planner model integration and other components. If any of those bring in large transitive dependencies (e.g. a big ML framework), we should try to use lighter-weight solutions (perhaps the Bedrock SDK is fairly light, so it might be fine). We will also enable init caching where possible – e.g., create global objects outside the handler. AWS notes that heavy static init contributes the most to cold start latency . By keeping our function stateless parts lean, we’ll complement the effects of PC (and for any cold start that does happen, minimize its impact).

**13. Monitor and Right-Size Timeouts:** The Retrieve Lambda currently has a 35s timeout. We aim never to hit that, but after optimizations, we may find our typical execution is, say, 2s. At that point, we could consider lowering the timeout to catch any hanging calls sooner (for instance, if Bedrock or Dynamo get stuck, we wouldn’t want to hang the full 35s). This is more of a safety tuning: we can set timeouts on external calls (Bedrock, Dynamo) via SDK configuration or use AWS Lambda Powertools to add timeouts to critical sections. This ensures a snappier failure in worst cases, improving system reliability. Also, adding **per-stage metrics** (as noted needed ) will help identify if any stage still occasionally slows down beyond expectation, so we can iterate on tuning.

By adjusting these infrastructure levers – memory, CPU architecture, concurrency, and package size – we effectively **remove any overheads the platform might be adding**. The Lambda will run at full throttle with minimal cold start interruptions. Together with the code and data improvements, this means the Lambda spends almost all its time doing only the **essential minimal work** for answering queries, and doing it on an optimized compute footprint.

## **Query Planning & Bedrock Interaction**

Although the focus is on retrieval performance, the **LLM-based planner** and **vector search** are also parts of the pipeline that we should ensure are efficient:

**14. Optimize LLM Query Planning:** The Planner (planner.py) uses a Bedrock-hosted Large Language Model to translate NL queries to structured filters. This step can contribute noticeable latency (each LLM call perhaps ~0.5–2 seconds depending on model). Several measures can improve this:

* **Cache and Reuse Plans:** As mentioned in caching, storing the last N query -> plan mappings in memory or DynamoDB can short-circuit the LLM for repeat questions. Even if phrasing differs, many user queries fall into a few categories (especially in enterprise search). We could implement a fuzzy match or normalization (e.g. sort filters and use them as key) so that “availabilities in Plano, Class A office” and “Office Class A availabilities located in Plano” hit the same cached plan. This is a bit complex (requires NLP normalization), but even exact matching on cleaned query text could yield hits. The key benefit is removing the model inference time entirely on cache hits.

* **LLM Model Choice and Prompt**: Evaluate if the current model is overkill for the domain. If a smaller model (or even a rules-based approach for simple queries) could be used, that would speed up responses. For instance, maybe 80% of queries are straightforward mappings that a deterministic logic (using the schema cache and a few heuristics) could handle. The system might already partially have this via *schema_decomposer* and *vocab_cache*. We could increase the reliance on those for low-complexity queries and only call the LLM when the query is ambiguous or complex. Additionally, ensure the prompt given to the LLM is concise and the response is parsed efficiently (the planner likely already does this well, given a ~500ms default timeout ). If not, fine-tuning or few-shot examples might reduce the need for lengthy reasoning by the model, thus faster replies.

* **Parallelize Planner with Data Fetch:** If the planner step is not cached and takes, say, 1–2s, we can overlap it with initial data retrieval in some cases. This is tricky because you normally need the plan before knowing what data to fetch. However, consider pre-fetching some generic data: e.g., start a vector search on the raw query text in parallel (without filters) while the planner works. If the planner fails or returns low confidence, we could fall back on that vector result (this is akin to a “hybrid search” fallback). If the planner succeeds with high confidence, its results likely supersede the generic search, but the vector search could still provide supplemental results. This approach is speculative and may not yield big latency savings unless the LLM is often the slowest part. It basically ensures we’re not idle during those 2 seconds. Another parallelism opportunity is *concurrent Bedrock requests*: if the planner were to possibly use two models or ask for clarification, doing those in parallel could save time, but that’s an edge case.

In practice, after implementing caching, the planner will be less of a bottleneck. We just need to ensure it’s not dominating the timeline for first-time queries. If it is, we allocate slightly more time or use a simpler model. The user experience is still acceptable if planning takes 1–2s as long as retrieval is fast, but our goal is a sub-5s total, so every second counts.

**15. Bedrock Vector Search Tuning:** The Knowledge Base (Bedrock vector store) is used to fetch relevant text chunks with metadata filters. To optimize its usage:

* **Use Metadata Filters Aggressively:** We already filter by record IDs or parent IDs to restrict vector results to the relevant subset . We should continue this practice for all cross-object queries so that Bedrock isn’t searching the entire vector index unnecessarily. This not only improves relevance but also performance, since searching within a smaller subset is faster. Bedrock’s managed vector search likely uses ANN (approximate nearest neighbor) algorithms which are quite fast, but filtering can add some overhead. If we notice vector search taking substantial time, consider reducing the number of chunks (topK) returned or using simpler queries. According to config, topK is ~20; we might try 10 if it suffices, to trim latency slightly (less data to transfer and re-rank).

* **Parallelize Vector and Graph**: Once we have the list of seed IDs from the graph traversal, the current pipeline then queries Bedrock with recordId IN seeds . We can actually kick off the vector search *as soon as we have those seeds, in parallel with enrichment*. Better, we could initiate the vector search even before enrichment is finished, since enrichment only adds metadata for the answer and doesn’t change which records we retrieve. Concretely, we could use Python threads: one thread performs the Bedrock KB query while another does the DynamoDB enrichment calls. These two subtasks do not depend on each other’s results. By doing them in parallel, we hide the latency of one behind the other. If each takes ~500ms, the sequential path would be 1s, but parallel could be ~500ms total (plus minor overhead). The final step would then merge the enriched fields into the vector results for the LLM. This requires careful synchronization but is straightforward.

* **Evaluate OpenSearch for Textual Queries:** The architecture mentioned an optional **Amazon OpenSearch Serverless (AOSS)** for hybrid search . If vector search latency or accuracy becomes an issue, we can consider using OpenSearch for certain queries (especially those that are keyword-heavy or require fast filtering on text fields). AOSS can perform full-text search with filters in perhaps single-digit millisecond latency for indexed data. For example, a query like “John Doe’s leases in 2020” might be easier to handle with a keyword search on a leases index (with date filters) than via vector similarity. We could maintain an OpenSearch index in parallel for key Salesforce objects (properties, availabilities, etc.) including their text and structured fields. The planner (or an intent classifier) could then route queries: if it’s a precise lookup or a term-based query, use OpenSearch; if it’s a natural language description or needs semantic match, use Bedrock vector. This kind of **hybrid retrieval** ensures we use the right tool for the job. It does introduce overhead of maintaining another index (and additional cost), so we should weigh it. But given AOSS was set up (perhaps not fully used yet), leveraging it could offload some queries entirely to a search service that can return results in tens of milliseconds. For instance, a direct term query in OpenSearch could return relevant IDs instantly, which we then enrich and answer. This is likely a medium priority optimization – not necessary to hit 5s latency, but useful for scaling and precision.

* **Bedrock Throughput Considerations:** If query volume increases, we should be mindful of Bedrock’s rate limits or scaling. We might implement pooling or reuse of Bedrock client connections for efficiency. Also, monitor the response time of Bedrock – if it becomes a bottleneck, we might consider **Amazon Kendra** or a self-hosted vector DB which we can better tune. At present, Bedrock’s vector store should be fine for our volume, but keeping an eye on it is part of performance best practices.

In essence, the search system should use *as little LLM “brainpower” and vector math as necessary* to satisfy the query. By parsing intelligently and using structure (graph or indices), we minimize the heavy-lifting that the LLM/vector components must do. When they are needed, we run them in parallel with other tasks to maximize throughput. This balanced approach ensures we meet latency targets while still delivering rich, AI-enhanced results.

## **Conclusion**

By implementing this multi-pronged optimization plan, the Salesforce graph-enhanced search system will become significantly faster and more scalable. In summary, we are **minimizing work per query** (through caching and precomputation), **making each unit of work execute faster** (through parallelism, more compute resources, and efficient code), and **avoiding unnecessary work altogether** (through smarter query planning and use of appropriate data stores). These changes directly target the known performance issues – for example, graph traversal will no longer “exceed 30s” as it did before  once batched and backed by pre-joined data. Instead, most queries should complete in a few seconds or less, and many common queries will be sub-second.

We have balanced quick wins (which yield immediate relief to users) with strategic improvements that ensure the system can handle growth in data and query complexity. Importantly, each recommendation has a clear rationale and follows industry best practices or AWS guidelines (whether it’s leveraging Lambda’s memory-to-CPU scaling , or using caches to avoid repeating expensive operations). We will also implement better monitoring (per-stage timing, cache hit rates, etc.) to continually verify the impact of each change and catch any regressions .

In terms of cost-benefit, most optimizations are either cost-neutral or even cost-saving for the performance gained – for example, a faster Lambda run often costs the same or less than a slow one . Where we add resources (like a cache or provisioned concurrency), we do so knowing they directly contribute to meeting the <2s latency goal that is critical for user experience.

Overall, this plan should **elevate the search experience to be instant and reliable**, turning the innovative graph-enhanced, AI-driven search from a working prototype into a production-grade solution. Adhering to these recommendations will provide Salesforce users with lightning-fast answers, even for complex cross-object questions, thereby fulfilling the promise of natural language search on their CRM data. The system will be not only faster but also more maintainable (with simpler query flows using precomputed data) and ready for future functionality enhancements.
