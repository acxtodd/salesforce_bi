### Issue 1: Stage Validation Inconsistency ✅ FIXED

**Reported By:** Code review feedback

**Problem Description:**
The Agentforce tool schemas (`create_opportunity_action.json` and `update_opportunity_stage_action.json`) defined Opportunity stage enums that did not match the backend validation in ActionEnablement metadata. This would cause server-side validation failures when Agentforce attempted to use stages like "Needs Analysis", "Value Proposition", "Perception Analysis", etc.

**Root Cause:**
The tool schemas were created with a comprehensive list of standard Salesforce Opportunity stages, but the ActionEnablement metadata (`ActionEnablement.Create_Opportunity.md-meta.xml` and `ActionEnablement.Update_Opportunity_Stage.md-meta.xml`) uses a simplified subset for the POC.

**Backend Validation:**
The Action Lambda validates inputs against the JSON schema stored in `ActionEnablement__mdt.InputSchemaJson__c` (see `lambda/action/index.py:1054-1067`).

**Allowed Stages:**

**create_opportunity:**
- Backend: `["Prospecting", "Qualification", "Proposal", "Negotiation"]`
- Tool schema (before fix): `["Prospecting", "Qualification", "Needs Analysis", "Value Proposition", "Id. Decision Makers", "Perception Analysis", "Proposal/Price Quote", "Negotiation/Review", "Closed Won", "Closed Lost"]`
- Tool schema (after fix): `["Prospecting", "Qualification", "Proposal", "Negotiation"]` ✅

**update_opportunity_stage:**
- Backend: `["Prospecting", "Qualification", "Proposal", "Negotiation", "Closed Won", "Closed Lost"]`
- Tool schema (before fix): `["Prospecting", "Qualification", "Needs Analysis", "Value Proposition", "Id. Decision Makers", "Perception Analysis", "Proposal/Price Quote", "Negotiation/Review", "Closed Won", "Closed Lost"]`
- Tool schema (after fix): `["Prospecting", "Qualification", "Proposal", "Negotiation", "Closed Won", "Closed Lost"]` ✅

**Changes Made:**

1. **salesforce/agentforce/create_opportunity_action.json:**
   - Updated `inputs.StageName.enum` to match backend: `["Prospecting", "Qualification", "Proposal", "Negotiation"]`
   - Updated description to clarify allowed stages
   - Updated examples to use valid stages

2. **salesforce/agentforce/update_opportunity_stage_action.json:**
   - Updated `inputs.StageName.enum` to match backend: `["Prospecting", "Qualification", "Proposal", "Negotiation", "Closed Won", "Closed Lost"]`
   - Updated description to clarify allowed stages
   - Updated examples to use "Negotiation" instead of "Negotiation/Review"
   - Updated `agentInstructions.stageProgression` to reflect actual progression: Prospecting → Qualification → Proposal → Negotiation → Closed Won/Lost

**Verification:**

### Issue 2: Confirmation Token Requirement ⚠️ DOCUMENTED (Not Fixed)

**Reported By:** Code review feedback

**Problem Description:**
The `/action` endpoint requires a `confirmationToken` field in every request (enforced in `lambda/action/index.py:129-135`). This token is currently generated by the Lightning Web Component (`salesforce/lwc/ascendixAiSearch/ascendixAiSearch.js:640-692`) as part of the two-step confirmation flow. Agentforce has no built-in mechanism to:
1. Display a preview modal to the user
2. Generate the confirmation token
3. Pass the token in the action request

**Root Cause:**
The Action Lambda was designed with the assumption that all action requests would come from the LWC component, which implements the two-step confirmation flow. The token serves as proof that the user saw and confirmed the preview.

**Current Behavior:**
Any direct Agentforce call to `/action` will fail with:
```
400 Bad Request: confirmationToken is required
```

**Why Not Fixed:**
This is an architectural limitation that requires a decision on the appropriate solution approach. The issue has been documented with three solution options, but implementing a fix requires choosing the right approach for the use case.

**Solutions Documented:**

**Option 1: Wrapper Flow (Recommended for POC)**
- Create Apex invocable wrapper that generates token server-side
- Agentforce calls wrapper instead of direct HTTP endpoint
- Wrapper calls `/action` with generated token
- Pros: No Lambda changes, maintains security model
- Cons: Extra layer of indirection
- Implementation: Full Apex code provided in README

**Option 2: Make Token Optional for Agentforce (Production)**
- Modify Lambda to accept `source` field
- Make token optional for `source: "agentforce"`
- Generate token server-side or skip validation
- Pros: Direct Agentforce integration, cleaner architecture
- Cons: Requires Lambda code changes, different security model
- Implementation: Pseudocode provided in README

**Option 3: Use LWC as Intermediary (Current POC)**
- Agentforce suggests actions but doesn't execute
- User opens LWC component to execute
- LWC displays preview and generates token
- Pros: No code changes, safest approach, already implemented
- Cons: Less seamless UX, requires manual step

**Recommendation:** Use Option 3 (LWC intermediary) for POC. Implement Option 2 (optional token) for production.

**Changes Made:**

1. **salesforce/agentforce/README.md:**
   - Added "⚠️ Important Limitation: Confirmation Token Requirement" section
   - Documented the problem in detail
   - Provided three solution options with implementation guidance
   - Added recommendation for POC vs. production
   - Updated registration instructions to note limitation
   - Added full Apex wrapper implementation code

2. **salesforce/agentforce/create_opportunity_action.json:**
   - Added note in `agentInstructions.responseFormat`: "NOTE: Due to confirmationToken requirement, this action currently requires the LWC component (ascendixAiSearch) to execute. Direct Agentforce invocation will fail until a wrapper is implemented."

3. **salesforce/agentforce/update_opportunity_stage_action.json:**
   - Added note in `agentInstructions.responseFormat`: "NOTE: Due to confirmationToken requirement, this action currently requires the LWC component (ascendixAiSearch) to execute. Direct Agentforce invocation will fail until a wrapper is implemented."

4. **salesforce/agentforce/TASK-21-IMPLEMENTATION-SUMMARY.md:**
   - Added "Known Limitations and Fixes Applied" section
   - Documented Issue 2 with full details
   - Listed all three solution options
   - Updated conclusion to note limitation
   - Changed status to "Complete with documented limitations"

**Impact:**
- Administrators are now aware of the limitation before attempting registration
- Three clear solution paths are provided
- POC can proceed using LWC component (Option 3)
- Production implementation has clear guidance (Option 2)
- No breaking changes to existing LWC functionality

**Future Work:**

- Implement Option 2 (optional token) for production deployment
