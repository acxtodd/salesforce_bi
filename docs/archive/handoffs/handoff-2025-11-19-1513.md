## 2. Blocking Issues

### Blocker A: Component-Level Metadata Errors
When attempting to deploy the metadata to the `ascendix-beta-sandbox`, the Metadata API returned 47 component failures. These require code/metadata remediation before deployment can succeed.

#### Critical Failures:
1.  **Custom Metadata Missing**:
    *   `ActionEnablement__mdt` type is missing or not deployed first.
    *   **Error**: "Custom metadata type ActionEnablement__mdt is not available in this organization."
    *   **Impact**: Blocks deployment of `ActionEnablement` records and Apex classes (`ActionEnablementController`, `AISearchBatchExport`) that reference this type.

2.  **Apex Compilation Errors**:
    *   `AISearchBatchExport`: References missing `AI_Search_Config__mdt` and undefined variables.
    *   `AISearchBatchExportScheduler`: Uses reserved keyword `sobject` as a variable name.
    *   `ActionEnablementController`: References missing methods/types.

3.  **LWC Compilation Errors**:
    *   `ascendixAiSearch`: `LWC1702: Invalid LWC imported identifier "createElement"`.
    *   `actionEnablementAdmin`: `LWC1060: Template expression doesn't allow Literal` (invalid usage of `{false}`).

4.  **Flow XML Duplication**:
    *   `Create_Opportunity_Flow`, `Update_Opportunity_Stage_Flow`, `Remove_AI_Agent_Actions_Permission_Set`.
    *   **Error**: "Element decisions is duplicated at this location in type Flow" / "Element screens is duplicated". This suggests invalid XML merging or corruption.

5.  **Named Credential Errors**:
    *   `Ascendix_RAG_API`: `authorizationHeaderName` is invalid in the metadata XML (likely unsupported in the target API version or NamedCredential type).

### Blocker B: Salesforce CLI / Environment Errors
When attempting to isolate deployments (e.g., deploying *only* Custom Metadata to fix Blocker A), the Salesforce CLI (`sf`) threw persistent environment errors:

*   **Command**: `sf project deploy start --source-dir customMetadata ...` (and variations)
*   **Error**: `Error (ENOENT): ENOENT: no such file or directory, scandir ''`
*   **Context**: This occurred even when using a `package.xml` manifest. This suggests the CLI is having trouble resolving paths within the non-standard `salesforce/` directory structure or there is a file system permissions/access issue in the current environment.

## 3. Attempted Resolutions

1.  **Full Deployment**: Failed due to "Too many files in zip" (Limit: 10,000). The `salesforce/` directory contains ~14,000 files (mostly node_modules/tests in LWC).
2.  **Targeted Directory Deployment**: Failed with the component errors listed in Blocker A.
3.  **Manual Fix & Granular Deployment**: Created missing `ActionEnablement__mdt` definition and tried to deploy just that folder to resolve dependencies. Failed with Blocker B (`ENOENT`).
4.  **Manifest Deployment**: Created `salesforce/manifest/package.xml` to deploy specific types. Failed with Blocker B (`ENOENT`).
