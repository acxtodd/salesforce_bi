```javascript
await waitForIndexAvailable(url, region, domain, indexName, 30);

async function waitForIndexAvailable(url, region, domain, indexName, maxAttempts) {
    for (let attempt = 1; attempt <= maxAttempts; attempt++) {
        const response = await sendRequest('GET', url, null, region, domain);
        if (response.statusCode === 200) {
            return true;
        }
        await new Promise(resolve => setTimeout(resolve, 2000));
    }
    throw new Error(`Index ${indexName} not available after ${maxAttempts} attempts`);
}
```

## Current Blocker: Index Creation Race Condition

### Problem Description

**Symptom**: Bedrock Knowledge Base creation fails with:
```
Resource handler returned message: "The knowledge base storage configuration provided 
is invalid... Dependency error document status code: 404, error message: no such index 
[salesforce-chunks]"
```

**Root Cause**: Race condition between index creation and Bedrock validation:

1. ✅ Custom resource invokes index-creator Lambda
2. ✅ Lambda successfully creates index in OpenSearch Serverless
3. ✅ Lambda logs: "Index salesforce-chunks created successfully"
4. ✅ Custom resource returns success to CloudFormation
5. ❌ CloudFormation immediately creates Bedrock Knowledge Base
6. ❌ Bedrock validates index existence → **404 Not Found**
7. ❌ Stack rolls back

**Timeline Evidence**:
```
12:54:46 PM | CREATE_COMPLETE | IndexResource (Custom Resource)
12:54:47 PM | CREATE_IN_PROGRESS | KnowledgeBase
12:54:48 PM | CREATE_FAILED | KnowledgeBase (404 - no such index)
```

**Gap**: ~1-2 seconds between index creation and Bedrock validation. OpenSearch Serverless needs propagation time before index is queryable.

### Why Wait Logic Didn't Help

**Attempt 1**: Added `waitForIndexAvailable()` function to index-creator Lambda
**Result**: Code not deployed in second attempt

**Reason**: CDK asset caching
- CDK calculates hash of Lambda code directory
- If hash unchanged, reuses existing S3 asset
- Even though `index.mjs` was modified, CDK didn't detect the change
- Ran `rm -rf cdk.out && npm run build && npx cdk synth` but deployment still used old Lambda

**Evidence**: Lambda logs from second deployment (1:27 PM) show same timestamps as first deployment (18:54), indicating Lambda code wasn't updated.

## Technical Analysis

### Why This Is Happening

1. **OpenSearch Serverless Eventual Consistency**:
   - Index creation is asynchronous in OpenSearch Serverless
   - Index metadata needs to propagate across the distributed system
   - Typically takes 2-10 seconds for index to be fully queryable

2. **CloudFormation Dependency Model**:
   - Custom resource reports success immediately after API call succeeds
   - CloudFormation proceeds to next resource (Knowledge Base) immediately
   - No built-in way to add "wait time" between resources

3. **Bedrock Validation Timing**:
   - Bedrock validates index existence during Knowledge Base creation
   - Validation happens before any data sync
   - Validation uses OpenSearch API (same path as Lambda)
   - But timing is too tight for eventual consistency

### Why Index Creator Shows Success

The Lambda successfully:
1. Creates the index via OpenSearch API
2. Receives 200 OK response
3. Logs "Index created successfully"
4. Returns success to CloudFormation

But "created" doesn't mean "queryable" in a distributed system. The index exists in the primary node but hasn't propagated to all query nodes yet.

## Possible Solutions

### Option 1: Remove Index Creator, Let Bedrock Create Index (RECOMMENDED)

**Approach**: 
- Remove the custom resource entirely
- Let Bedrock Knowledge Base create the index automatically
- Bedrock has built-in retry logic for index creation

**Pros**:
- Eliminates race condition
- Simpler architecture
- Bedrock handles eventual consistency internally
- Less custom code to maintain

**Cons**:
- Less control over index configuration
- Bedrock uses default index settings
- May not support custom vector field configurations

**Implementation**:
```typescript
// In lib/search-stack.ts, remove:
// - IndexCreator Lambda
// - IndexProvider
// - IndexResource custom resource
// - indexResource dependency from knowledgeBase

// Keep everything else the same
```

**Risk**: Unknown if Bedrock auto-creates index with correct vector field configuration (1024 dimensions for Titan v2).

### Option 2: Add Explicit Wait in Custom Resource

**Approach**:
- Fix CDK caching issue
- Deploy updated index-creator with wait logic
- Verify index is queryable before returning success

**Pros**:
- Full control over index configuration
- Explicit validation before proceeding
- Clear error messages if index creation fails

**Cons**:
- More complex
- Requires fixing CDK caching
- May need longer timeout (current: 5 minutes)

**Implementation**:
1. Force CDK to rebuild Lambda asset:
   ```bash

   touch lambda/index-creator/index.mjs
   
   # Option B: Clear CDK cache
   rm -rf cdk.out
   rm -rf node_modules/.cache
   
   # Option C: Change Lambda configuration
   # Update timeout or memory in search-stack.ts
   ```

2. Redeploy with updated code

3. Verify wait logic executes in CloudWatch Logs

**Risk**: Even with wait logic, there might be a different propagation path for Bedrock vs Lambda.

### Option 3: Add Delay Between Resources

**Approach**:
- Use CDK custom resource with sleep/delay
- Insert between IndexResource and KnowledgeBase
- Wait 10-30 seconds before allowing KB creation

**Pros**:
- Simple to implement
- Guaranteed wait time
- No changes to index-creator logic

**Cons**:
- Arbitrary wait time (might be too short or too long)
- Adds deployment time
- Doesn't validate index is actually ready

**Implementation**:
```typescript
// Create a delay custom resource
const delayResource = new cdk.CustomResource(this, 'IndexDelay', {
  serviceToken: delayProvider.serviceToken,
  properties: {
    delaySeconds: 30
  }
});

delayResource.node.addDependency(indexResource);
this.knowledgeBase.node.addDependency(delayResource);
```

**Risk**: Arbitrary delay might not be sufficient for all cases.

### Option 4: Use Bedrock Data Source Sync Instead

**Approach**:
- Create Knowledge Base without data source initially
- Let KB create index
- Add data source after KB is active
- Trigger sync manually or via Lambda

**Pros**:
- Separates KB creation from index validation
- Bedrock handles index creation timing
- More control over sync process

**Cons**:
- More complex deployment
- Requires additional orchestration
- Data source must be added post-deployment

**Implementation**: Requires multi-stage deployment or additional automation.

## Key Learnings

### 1. OpenSearch Serverless vs Managed Cluster

**Discovery**: Bedrock Knowledge Bases have different requirements for different OpenSearch types:
- **Managed Cluster**: Requires public endpoint (incompatible with VPC-only architecture)
- **Serverless**: Supports private VPC endpoints via network policies

**Implication**: OpenSearch Serverless is the correct choice for private deployments, not a fallback.

### 2. Network Policy Requirements

**Discovery**: Private OpenSearch Serverless collections require TWO access paths:
- `SourceVPCEs`: For Lambda/application access through VPC endpoint
- `SourceServices`: For AWS service access (Bedrock, etc.)

**Implication**: Both must be specified or Bedrock cannot access the collection.

### 3. IAM Permission Granularity

**Discovery**: OpenSearch Serverless requires explicit permissions at multiple levels:
- Control plane: `aoss:APIAccessAll`, `aoss:DashboardsAccessAll`
- Collection level: `aoss:CreateCollectionItems`, `aoss:UpdateCollectionItems`
- Index level: `aoss:ReadDocument`, `aoss:WriteDocument`

**Implication**: Generic `aoss:APIAccessAll` is not sufficient for data operations.

### 4. CDK Asset Caching

**Discovery**: CDK caches Lambda code assets based on directory hash:
- Changing file content doesn't always trigger rebuild
- `cdk.out/` removal doesn't clear asset cache
- May need to change Lambda configuration to force update

**Implication**: Need reliable method to force Lambda code updates during development.

### 5. Eventual Consistency in Distributed Systems

**Discovery**: OpenSearch Serverless index creation is eventually consistent:
- API returns success immediately
- Index takes 2-10 seconds to be fully queryable
- No synchronous "wait for ready" API

**Implication**: Need explicit wait/retry logic or architectural changes to handle this.
