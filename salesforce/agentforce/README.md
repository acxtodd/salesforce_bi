# Agentforce Tool Registration Guide

This directory contains tool schema definitions for integrating the Ascendix AI Search RAG system with Salesforce Agentforce.

## Overview

Agentforce is Salesforce's AI agent platform that can invoke external tools to enhance its capabilities. By registering these tools, Agentforce agents can:

1. **Search Salesforce data** using the `retrieve_knowledge` tool
2. **Generate grounded answers** using the `answer_with_grounding` tool

Both tools enforce Salesforce sharing rules and field-level security, ensuring agents only access data the user has permission to see.

## Prerequisites

Before registering these tools, ensure:

- [ ] Agentforce is enabled in your Salesforce org
- [ ] The Ascendix RAG API is deployed and accessible via PrivateLink
- [ ] Named Credential `Ascendix_RAG_API` is configured with:
  - Private API Gateway endpoint URL
  - API key authentication
  - Proper network connectivity via Salesforce Private Connect

## ⚠️ Important Limitation: Confirmation Token Requirement

**CRITICAL:** The Phase 2 agent actions (`create_opportunity` and `update_opportunity_stage`) currently have a limitation that prevents direct use from Agentforce:

### The Problem

The `/action` endpoint requires a `confirmationToken` field in every request (see `lambda/action/index.py:129-135`). This token is currently generated by the Lightning Web Component (`salesforce/lwc/ascendixAiSearch/ascendixAiSearch.js:640-692`) as part of the two-step confirmation flow. However, Agentforce has no built-in mechanism to:

1. Display a preview modal to the user
2. Generate the confirmation token
3. Pass the token in the action request

As a result, any direct Agentforce call to these actions will fail with:
```
400 Bad Request: confirmationToken is required
```

### Solutions

Choose one of the following approaches:

#### Option 1: Wrapper Flow (Recommended for POC)

Create an Apex-invocable wrapper that Agentforce can call directly:

1. **Create Apex class:** `AgentforceActionWrapper.cls`
   ```apex
   public class AgentforceActionWrapper {
       @InvocableMethod(label='Execute Agent Action' description='Wrapper for Agentforce to execute actions with auto-generated token')
       public static List<ActionResult> executeAction(List<ActionRequest> requests) {
           ActionRequest req = requests[0];
           
           // Generate confirmation token server-side
           String token = generateToken(req.actionName, req.inputs);
           
           // Call Action Lambda via Named Credential
           HttpRequest httpReq = new HttpRequest();
           httpReq.setEndpoint('callout:Ascendix_RAG_API/action');
           httpReq.setMethod('POST');
           httpReq.setHeader('Content-Type', 'application/json');
           
           Map<String, Object> payload = new Map<String, Object>{
               'actionName' => req.actionName,
               'inputs' => req.inputs,
               'salesforceUserId' => UserInfo.getUserId(),
               'confirmationToken' => token
           };
           httpReq.setBody(JSON.serialize(payload));
           
           Http http = new Http();
           HttpResponse res = http.send(httpReq);
           
           // Parse and return result
           ActionResult result = new ActionResult();
           if (res.getStatusCode() == 200) {
               Map<String, Object> body = (Map<String, Object>)JSON.deserializeUntyped(res.getBody());
               result.success = (Boolean)body.get('success');
               result.recordIds = (List<String>)body.get('recordIds');
           } else {
               result.success = false;
               result.error = res.getBody();
           }
           return new List<ActionResult>{ result };
       }
       
       private static String generateToken(String actionName, Map<String, Object> inputs) {
           Long timestamp = System.currentTimeMillis();
           String data = actionName + JSON.serialize(inputs) + timestamp;
           Blob hash = Crypto.generateDigest('SHA-256', Blob.valueOf(data));
           return 'token_' + EncodingUtil.convertToHex(hash).substring(0, 16) + '_' + timestamp;
       }
       
       public class ActionRequest {
           @InvocableVariable(required=true)
           public String actionName;
           @InvocableVariable(required=true)
           public Map<String, Object> inputs;
       }
       
       public class ActionResult {
           @InvocableVariable
           public Boolean success;
           @InvocableVariable
           public List<String> recordIds;
           @InvocableVariable
           public String error;
       }
   }
   ```

2. **Register wrapper in Agentforce:**
   - Use the Apex invocable method instead of direct HTTP calls
   - Agentforce calls `AgentforceActionWrapper.executeAction`
   - Wrapper generates token and calls `/action` endpoint

#### Option 2: Make Token Optional for Agentforce (Production Approach)

Modify the Action Lambda to make `confirmationToken` optional when called from Agentforce:

1. **Update Lambda validation:**
   ```python
   # In lambda/action/index.py, modify _parse_request:
   
   confirmation_token = payload.get("confirmationToken")
   source = payload.get("source", "unknown")  # Add source field
   
   # Only require token for LWC calls, not Agentforce
   if source == "lwc" and not confirmation_token:
       raise ValidationError("confirmationToken is required for LWC calls")
   
   # For Agentforce, generate token server-side or skip validation
   if not confirmation_token:
       confirmation_token = _generate_server_side_token(action_name, inputs)
   ```

2. **Update tool schemas:**
   - Make `confirmationToken` optional in input schema
   - Add `source: "agentforce"` field to identify caller
   - Document that Agentforce bypasses two-step confirmation

3. **Security considerations:**
   - Agentforce has its own confirmation UI
   - Rate limiting still applies
   - Audit logging still captures all actions
   - Consider requiring admin approval for Agentforce actions

#### Option 3: Use LWC as Intermediary (Current POC Approach)

Keep the current implementation and use the LWC component as the action interface:

1. **Agentforce suggests actions** but doesn't execute them
2. **User opens LWC component** to execute suggested actions
3. **LWC displays preview modal** and generates confirmation token
4. **LWC calls `/action` endpoint** with token

This is the safest approach but provides a less seamless user experience.

### Recommendation

For the POC, use **Option 3** (LWC intermediary) since it's already implemented and tested. For production, implement **Option 2** (make token optional for Agentforce) with proper security controls.

### Updated Registration Instructions

Given this limitation, the registration steps below are for **documentation purposes only**. The actions cannot be directly invoked by Agentforce until one of the solutions above is implemented. Instead, use the LWC component (`ascendixAiSearch`) to execute actions with proper two-step confirmation.

## Tool Schemas

### Phase 1: Read-Only Search and Answer

#### 1. retrieve_knowledge

**File:** `retrieve_knowledge_tool.json`

**Purpose:** Searches across indexed Salesforce records using hybrid semantic and keyword search.

**Key Features:**
- Searches Account, Opportunity, Case, Note, Property__c, Lease__c, Contract__c
- Supports filters by object type, region, business unit, quarter
- Returns relevant chunks with metadata and presigned preview URLs
- Enforces sharing rules and field-level security

**When to Use:**
- Finding specific information across Salesforce records
- Answering questions that require searching multiple object types
- As a first step before generating answers to ensure grounding

#### 2. answer_with_grounding

**File:** `answer_with_grounding_tool.json`

**Purpose:** Generates natural language answers with paragraph-level citations.

**Key Features:**
- Uses retrieval-augmented generation (RAG) for grounded responses
- Streams answers using Server-Sent Events (SSE)
- Includes citations to source Salesforce records
- Applies Bedrock Guardrails for safety and grounding validation
- Supports multi-turn conversations via sessionId

**When to Use:**
- Providing comprehensive answers to user questions
- Synthesizing information from multiple records
- When citations and source attribution are important

### Phase 2: Agent Actions (Create/Update)

#### 3. create_opportunity

**File:** `create_opportunity_action.json`

**Purpose:** Creates a new Opportunity record with validation and two-step confirmation.

**Key Features:**
- Creates Opportunity with Name, Account, Amount, CloseDate, Stage, Description
- Requires two-step confirmation (preview + confirm)
- Enforces CRUD permissions and sharing rules
- Rate limited (default: 20 creates per user per day)
- Logs all executions to AI_Action_Audit__c

**When to Use:**
- When user explicitly requests to create a new Opportunity
- When user provides sufficient information for required fields
- As a follow-up action after searching for an Account

**Required Fields:**
- Name (string, 1-120 chars)
- AccountId (18-char ID starting with 001)
- CloseDate (date in YYYY-MM-DD format)
- StageName (valid picklist value)

**Optional Fields:**
- Amount (number, USD)
- Description (string, up to 32000 chars)

#### 4. update_opportunity_stage

**File:** `update_opportunity_stage_action.json`

**Purpose:** Updates an existing Opportunity's stage and probability with validation and two-step confirmation.

**Key Features:**
- Updates Opportunity stage and probability
- Requires two-step confirmation (preview + confirm)
- Validates user has edit access to the Opportunity
- Enforces CRUD permissions and sharing rules
- Rate limited (default: 50 updates per user per day)
- Uses optimistic concurrency control
- Logs all executions to AI_Action_Audit__c

**When to Use:**
- When user explicitly requests to update an Opportunity stage
- When user wants to move a deal forward in the sales process
- When user wants to mark an Opportunity as won or lost

**Required Fields:**
- OpportunityId (18-char ID starting with 006)
- StageName (valid picklist value)

**Optional Fields:**
- Probability (integer, 0-100)

### Step 6: Configure Agent to Use Tools

1. Navigate to **Setup** → **Agentforce** → **Agents**
2. Select or create an agent
3. In the **Actions** section, add all tools:
   - `retrieve_knowledge` (Phase 1)
   - `answer_with_grounding` (Phase 1)
   - `create_opportunity` (Phase 2)
   - `update_opportunity_stage` (Phase 2)

4. **Configure Tool Priority:**
   - Set `answer_with_grounding` to prefer retriever grounding over model priors
   - Configure the agent to use `retrieve_knowledge` first for factual questions
   - Set up tool chaining: retrieve → answer
   - Configure actions to require preview + confirmation

5. **Add Instructions:**
   ```
   When answering questions about Salesforce data:
   1. Always use retrieve_knowledge first to find relevant information
   2. Use answer_with_grounding to generate responses with citations
   3. Prefer grounded answers over generic knowledge
   4. Always include the user's Salesforce User ID in tool calls
   5. If no accessible results are found, inform the user politely
   
   When executing actions (Phase 2):
   1. Always show preview before executing actions
   2. Only suggest actions when user explicitly requests to create or update records
   3. Gather all required information before proposing an action
   4. Use natural language to describe what will be created/updated in the preview
   5. After successful execution, provide a link to the created/updated record
   6. Handle rate limits and errors gracefully with friendly messages
   ```

6. **Save** the agent configuration

## Configuration Best Practices

### Grounding Strategy

Configure Agentforce to prefer retriever grounding over model priors:

1. **Tool Priority:** Set `answer_with_grounding` as high priority for Salesforce questions
2. **Fallback Behavior:** Only use generic model knowledge if no Salesforce data is found
3. **Citation Requirements:** Always require citations for factual claims
4. **Validation:** Validate that cited records are in the retrieved context

### Security Configuration

1. **User ID Mapping:**
   - Ensure `salesforceUserId` is automatically populated with the current user's ID
   - Never allow users to specify a different user ID
   - Validate user ID format (15 or 18 characters, starts with "005")

2. **API Key Security:**
   - Use separate API keys for user requests vs. service requests
   - Rotate API keys every 90 days
   - Monitor API key usage for anomalies

3. **Rate Limiting:**
   - Configure appropriate rate limits in API Gateway
   - Monitor usage per user and per agent
   - Set up alerts for unusual activity

### Performance Optimization

1. **Caching:**
   - Enable authorization context caching (24-hour TTL)
   - Enable retrieval results caching (30-120 second TTL)
   - Monitor cache hit rates (target >95% for authZ, >60% for retrieval)

2. **Timeout Configuration:**
   - Set appropriate timeouts for tool calls (recommend 30 seconds)
   - Configure retry logic for transient failures
   - Implement graceful degradation if tools are unavailable

3. **Monitoring:**
   - Track tool invocation counts and latency
   - Monitor error rates by error type
   - Set up alerts for performance degradation

## Troubleshooting

### Tool Not Appearing in Agent

**Symptoms:** Tool is registered but not available in agent configuration

**Solutions:**
1. Verify Named Credential is active and accessible
2. Check that tool is marked as "Active" in Agent Actions
3. Verify user has permission to use the tool
4. Refresh the agent configuration page

### Authorization Errors

**Symptoms:** "Access Denied" or "403 Forbidden" errors

**Solutions:**
1. Verify `salesforceUserId` is being passed correctly
2. Check that Named Credential has valid API key
3. Verify PrivateLink connectivity is working
4. Test API endpoint directly using Postman or curl

### No Results Returned

**Symptoms:** Tool returns empty results or "no accessible results"

**Solutions:**
1. Verify user has sharing rule access to the queried objects
2. Check that objects are indexed in the RAG system
3. Verify CDC pipeline is running and data is fresh
4. Test with a user who has broader access (e.g., System Administrator)

### Slow Performance

**Symptoms:** Tool calls take longer than expected

**Solutions:**
1. Check CloudWatch metrics for Lambda cold starts
2. Verify provisioned concurrency is configured
3. Check OpenSearch cluster health and performance
4. Review cache hit rates and optimize caching strategy
5. Consider reducing `topK` parameter for faster retrieval

### Streaming Issues (answer_with_grounding)

**Symptoms:** Answers not streaming or appearing all at once

**Solutions:**
1. Verify Agentforce supports SSE streaming
2. Check Named Credential timeout settings
3. Test streaming endpoint directly
4. Consider using non-streaming mode if SSE is not supported

### Action Enablement Metadata

- **FlowName__c:** Create_Opportunity_Flow
- **InputSchemaJson__c:** JSON schema for validation
- **OutputSchemaJson__c:** JSON schema for response

**update_opportunity_stage:**
- **Enabled__c:** true (can be disabled by admin)

- **FlowName__c:** Update_Opportunity_Stage_Flow
- **InputSchemaJson__c:** JSON schema for validation
- **OutputSchemaJson__c:** JSON schema for response

### Kill Switch

Administrators can immediately disable actions:

1. **Via Metadata:**
   - Navigate to **Setup** → **Custom Metadata Types** → **ActionEnablement**
   - Edit the record for the action (e.g., create_opportunity)
   - Set **Enabled__c** = false
   - Save (takes effect immediately)

2. **Via Permission Sets:**
   - Remove AI_Agent_Actions_Editor permission set from users
   - Users will lose Create/Edit permissions on target objects
   - Actions will fail with permission errors

3. **Via Agentforce:**
   - Navigate to **Setup** → **Agentforce** → **Agent Actions**
   - Deactivate the action
   - Agent will no longer offer the action to users
